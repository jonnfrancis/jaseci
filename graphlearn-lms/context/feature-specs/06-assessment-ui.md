We're implementing the assessment-taking experience.

This feature is responsible for rendering assessments generated by the LMS and collecting learner responses.

The UI must remain completely separate from assessment generation, scoring, mastery calculation, and roadmap logic.

Do not place business logic inside React components.

---

### Create Assessment UI Module

Create a dedicated assessment UI feature.

Suggested structure:

```text
features/
└── assessment/
    ├── components/
    ├── hooks/
    ├── types/
    └── pages/
```

Keep UI concerns isolated from walkers and graph logic.

---

### Create Assessment Page

Create an assessment-taking page.

Responsibilities:

* load assessment data
* display assessment title
* display assessment description
* display progress
* render questions
* collect learner responses
* submit completed assessment

Do not perform evaluation.

Do not calculate scores.

Do not generate recommendations.

---

### Create Reusable Assessment Components

Create reusable assessment components.

Examples:

* AssessmentContainer
* AssessmentHeader
* AssessmentProgress
* AssessmentQuestion
* AssessmentQuestionList
* AssessmentNavigation
* AssessmentSubmitDialog

Components should be presentation-focused.

Avoid embedding business logic.

---

### Support Multiple Question Types

Support rendering at minimum:

* multiple choice
* short answer

Multiple choice should:

* display available options
* allow a single selection

Short answer should:

* display a textarea
* support free-form responses

Design components so future question types can be added without major refactoring.

Future question types may include:

* multi-select
* ordering
* matching
* coding challenges

Do not implement them now.

---

### Create Assessment State Management

Create a dedicated assessment state layer.

Responsibilities:

* store responses
* update responses
* validate required responses
* track completion progress

The UI should not directly mutate assessment entities.

Use a centralized state pattern.

Examples:

* React state
* context
* custom hook

Choose the simplest approach that fits the project architecture.

---

### Create Assessment Submission Flow

Implement assessment submission.

Responsibilities:

1. Validate required responses

2. Build submission payload

3. Submit responses

4. Show submission success state

Do not score responses.

Do not calculate mastery.

Do not generate a roadmap.

Do not invoke evaluation walkers.

Submission should only persist learner answers.

---

### Create Assessment Response Models

Create UI types for learner responses.

Examples:

```ts
type AssessmentResponse

type QuestionResponse

type MultipleChoiceResponse

type ShortAnswerResponse
```

These should remain separate from graph entities.

Do not expose graph implementation details to the UI.

---

### Create Progress Tracking

Display learner progress.

Examples:

* current question count
* answered questions count
* completion percentage

Progress should update immediately as responses change.

Progress should not depend on server round trips.

---

### Create Validation

Before submission:

* required questions must be answered
* invalid responses should be highlighted
* submission should be blocked until validation passes

Validation should be user-friendly.

Avoid disruptive error messages.

---

### Loading and Error States

Provide UI states for:

* loading assessment
* assessment unavailable
* submission in progress
* submission success
* submission failure

The user should always understand the current assessment state.

---

### Design Requirements

Use existing design system components.

Use:

* Button
* Card
* Dialog
* Input
* Textarea
* ScrollArea

Maintain consistency with the existing dark theme.

Do not introduce light-theme styling.

Do not create duplicate UI primitives.

---

### Accessibility

Ensure:

* keyboard navigation works
* labels are associated with inputs
* focus states are visible
* screen readers can identify questions and answers

Accessibility should be built into components from the start.

---

### Future Compatibility

Design this UI so future features can reuse it.

Future workflows include:

* scored assessments
* challenge completion
* lesson checkpoints
* mastery verification
* certification exams

This assessment UI should become the foundation for all learner evaluation experiences.

Do not implement those workflows now.

---

### Check when done

* Assessment page renders successfully
* Questions display correctly
* Multiple choice questions work
* Short answer questions work
* Responses are collected correctly
* Validation prevents incomplete submissions
* Assessment can be submitted
* Submission persists responses
* Progress tracking updates correctly
