# 50.5-course-blueprint-ui.md

## Overview

Connect the existing lecturer frontend into one reliable course-building journey:

```text
verified source material
→ blueprint generation
→ generation progress and recovery
→ lecturer review and editing
→ approval
→ publication readiness
→ immutable publication
→ learner-integration handoff
```

Features 44.5 and 45–50 already provide most backend contracts and several frontend routes. This feature must extend those integrations rather than create a second blueprint or publication workflow.

The core rule is:

```text
The backend owns lifecycle truth.
The course workspace explains that truth and offers exactly the next authorized action.
```

# Goals

Implement a cohesive UI that:

1. Connects verified source readiness to blueprint generation.
2. Shows durable generation progress after navigation or refresh.
3. Supports generation retry and deliberate full regeneration.
4. Routes eligible generated blueprints into lecturer review.
5. Surfaces validation, evidence, revision, save, and approval state.
6. Connects approved blueprints to publication readiness.
7. Requires explicit confirmation before immutable publication.
8. Shows active published version and publication history.
9. Supports creating a new draft version from a published course.
10. Keeps existing learners pinned to historical versions.
11. Gives every backend state one clear lecturer-facing explanation and next action.
12. Restores the same workflow state after refresh, restart, or another device login.
13. Keeps authorization and readiness decisions on the server.
14. Preserves the existing course workspace tabs and specialist routes.
15. Produces a stable published-course handoff for learner-facing catalogue, enrollment, assessment, and roadmap subsystems.

---

# Non-goals

This feature does not:

- implement blueprint generation, graph generation, validation, review persistence, or publication persistence;
- introduce another blueprint or publication service;
- publish automatically after approval;
- implement the learner catalogue or enrollment UI;
- open enrollment automatically;
- generate diagnostic assessments, roadmaps, lessons, or challenges;
- move existing enrollments to a newer version;
- expose raw source documents, storage paths, prompts, or AI execution payloads;
- infer readiness from local timers, route history, or button clicks;
- allow direct mutation of published curriculum;
- silently regenerate lecturer-edited content.

# Frontend architecture

Use a stateful-shell architecture:

```text
CourseWorkspacePage
├── owns course + workflow projection
├── refreshes backend state
├── renders persistent lifecycle stepper
├── renders Overview
├── renders Course setup
├── renders Source material
├── renders Curriculum handoff
└── renders Publishing handoff

Specialist routes
├── CourseInterpretationPage: generation execution/progress
├── BlueprintReviewPage: inspection/edit/approval
└── CoursePublicationPage: readiness/publish/history
```

The shell and specialist pages must use the same backend-derived workflow projection. They must not independently invent state labels from unrelated fields.

# Unified workflow projection

Add or extend a typed private read endpoint:

```text
get_course_build_workflow(track_id, track_version_id)
```

Return:

```jac
obj CourseBuildWorkflowView {
    has track_id: str;
    has track_version_id: str;

    has course_status: str;
    has version_status: str;
    has active_version_id: str | None;
    has version_number: int;

    has current_stage: CourseBuildStage;
    has stage_label: str;
    has progress_percent: int;
    has next_action: CourseBuildAction;

    has source: CourseSourceWorkspaceView;
    has blueprint: BlueprintGenerationStatusView;
    has review: CourseBuildReviewSummary | None;
    has publication: PublicationReadinessResult | None;

    has blockers: list[CourseBuildBlocker];
    has warnings: list[str];

    has can_review: bool;
    has can_approve: bool;
    has can_publish: bool;
    has can_create_new_version: bool;

```

Supporting enums:

```jac

    SOURCE_REQUIRED,
    SOURCE_PROCESSING,
    SOURCE_FAILED,

    GENERATING_BLUEPRINT,

    VALIDATING_BLUEPRINT,

    APPROVED,
    PUBLICATION_BLOCKED,
    READY_TO_PUBLISH,
    PUBLISHING,
    PUBLISHED,
    SUPERSEDED

    COMPLETE_SETUP,
    UPLOAD_SOURCE,
    WAIT_FOR_SOURCE,
    RETRY_SOURCE,

    WAIT_FOR_GENERATION,
    RETRY_GENERATION,
```

```jac
obj CourseBuildBlocker {
    has code: str;
    has stage: CourseBuildStage;
    has message: str;
    has remediation_action: CourseBuildAction;
    has retryable: bool = False;
}
```

# Projection rules

The backend projection must resolve state in this precedence order:

```text
published/active version
publication in progress or failed
approved review
review changes required/in progress
validated blueprint ready for review
blueprint validation/generation running or failed
source processing running or failed
source readiness
course setup
```

Terminal or blocking backend states override optimistic client state.

Examples:

```text
client clicked Generate + backend says QUEUED
→ GENERATING_BLUEPRINT

client remembers review route + active blueprint is superseded
→ route back to current workflow action

client clicked Publish + readiness is stale
→ PUBLICATION_BLOCKED with backend blocking code
```

# Blueprint generation page

Extend `CourseInterpretationPage.cl.jac`; do not create another generation page.

## Entry guard

Allow entry when any of these are true:

```text
source is interpretation eligible
or blueprint generation is already running
or generation failed and is retryable
or a generated blueprint exists
```

This permits refresh/recovery without requiring the source workspace to remain in its original transient state.

## Page states

### Ready

Show:

- verified source filename and checksum-safe summary;
- what will be generated;
- grounding statement;
- expected workflow after generation;
- Generate blueprint button.

### Queued/running

Use `get_course_blueprint_generation_status` as authoritative.

Show:

- current human-readable step;
- progress percentage;
- warning count;
- durable background-processing explanation;
- Leave page safely action.

Polling:

```text
2 seconds while status is active
back off to 5 seconds after 30 seconds
stop on terminal state
stop on unmount
```

### Generated/validated

Show counts and confidence when available, then route to review only when backend review eligibility is true.

Primary action:

```text
Review generated blueprint
```

### Failed

Show safe error mapping and `retry_available`.

Actions:

```text
Retry generation
Replace source material
Back to course
```

### Regeneration

If an active blueprint or lecturer edits exist, generation must not reuse the first-generation button.

Use an explicit confirmation dialog describing:

```text
new blueprint generation may supersede the current review
manual changes will not be silently overwritten
publication approval will become stale
```

## Editing

Use the existing `update_blueprint_entity` endpoint.

Support only backend allowlisted fields.


Rules:

- debounce text saves only after local validation;
- send the last known `expected_revision`;
- replace local review state with the server response;
- never increment revision locally;
- never discard failed edits;
- on conflict, stop autosave and offer Refresh current / Copy my changes;
- show field-level error text for non-editable or invalid fields.

# Error mapping

Create one client helper for these workflows:

```text
lib/course_build/course_build_errors.cl.jac
```

Map backend codes to safe messages and actions.

Examples:

| Code | Message | Action |
|---|---|---|
| `BLUEPRINT_SOURCE_NOT_READY` | Source processing must finish before generation. | View source |
| `BLUEPRINT_GENERATION_ALREADY_RUNNING` | Blueprint generation is already running. | View progress |
| `BLUEPRINT_REVIEW_REVISION_CONFLICT` | This blueprint changed in another session. | Refresh review |
| `BLUEPRINT_REVIEW_APPROVAL_BLOCKED` | Resolve blocking validation issues before approval. | View issues |
| `COURSE_PUBLICATION_APPROVAL_STALE` | The course changed after approval. Approve the latest revision again. | Return to review |
| `COURSE_PUBLICATION_SOURCE_STALE` | The approved source no longer matches this version. | View source |
| `COURSE_PUBLICATION_VERSION_CONFLICT` | Publication readiness changed. Refresh before publishing. | Refresh readiness |
| `COURSE_VERSION_IMMUTABLE` | Published curriculum cannot be edited. Create a new version. | Create version |

Never show:

```text
EXECUTION_ERROR
stack trace
raw exception
storage key
prompt payload
```

# Refresh and recovery

Every page must recover from direct navigation and refresh.

On mount:

```text
load course
resolve authoritative draft/active version
load unified workflow projection
render backend state
```

Do not require prior page-local state.

When an operation finishes:

```text
replace local operation state from returned DTO
then refresh unified workflow projection
```

# Learner Integration readiness boundary

Feature 50.5 stops at a published, active, immutable version. It must expose a clear handoff state, not implement learner behavior.

## Ready-for-learner-integration definition

```text
track.status == PUBLISHED
active_version_id is not None
active version.status == PUBLISHED
CoursePublication snapshot exists
publication state audit is valid
curriculum graph belongs to active version
visibility is known
enrollment-open is explicit
```

## Handoff projection

Add or extend a server-owned projection usable by the next subsystem:

```jac
obj LearnerIntegrationHandoffView {
    has ready: bool;
    has track_id: str;
    has track_version_id: str | None;
    has version_number: int | None;
    has publication_id: str | None;

    has title: str;
    has short_description: str;
    has course_code: str | None;
    has subject_area: str | None;
    has academic_level: str | None;
    has visibility: str;
    has enrollment_open: bool;

    has module_count: int;
    has skill_count: int;
    has outcome_count: int;

    has blocking_codes: list[str];
}
```

This projection is informational for the lecturer UI and becomes the contract boundary for:

```text
published course catalogue
course enrollment
version-pinned diagnostic assessment
version-pinned roadmap generation
```

It must not expose draft blueprint details or lecturer review notes.

- Blocking errors use an alert region.
- Tabs and dialogs use existing accessible primitives.
- Confidence and severity are not conveyed by color alone.
- Reorder controls include accessible labels and current positions.
- Focus returns to the triggering control when dialogs close.
- After route transitions, focus moves to the page heading.
- Respect reduced-motion preferences.

# Testing strategy

## Workflow projection tests

Cover every `CourseBuildStage` and precedence conflicts.

Examples:

```text
source ready + no blueprint → READY_FOR_GENERATION
generation running → GENERATING_BLUEPRINT
valid blueprint → READY_FOR_REVIEW
approved review + stale validation → PUBLICATION_BLOCKED
published active snapshot → PUBLISHED
```

## Generation UI tests

- eligible source enables generation;
- direct refresh resumes running generation;
- retry appears only when backend permits;
- generated blueprint routes to review;
- regeneration requires confirmation.

## Review UI tests

- modules, concepts, outcomes, prerequisites render;
- selected entity evidence renders safely;
- allowlisted edit sends current revision;
- returned revision replaces local revision;
- conflict preserves unsaved text;
- blocking issue disables approval;
- approval makes editors read-only;
- approved state links to publication.

## Publication UI tests

- readiness blockers map to safe messages;
- publish requires confirmation;
- duplicate clicks send one logical request;
- unknown request outcome reloads history;
- published confirmation shows immutable version;
- create-new-version routes to the new draft;
- publication does not imply enrollment open.

## Refresh tests

Open every specialist route directly with no prior client state.

## Authorization tests

- owner lecturer sees allowed actions;
- other lecturer receives safe not-found/denial behavior;
- learner cannot call lecturer RPCs;
- suspended lecturer is denied;
- admin controls appear only from explicit permissions.

## Learner handoff tests

- draft is never learner-ready;
- approved but unpublished is never learner-ready;
- published active version with valid snapshot is ready;
- enrollment-closed remains ready for integration but not open for enrollment;
- superseded version is not selected for new learners;
- no draft/review notes leak through the handoff projection.

# Expected result

After this specification is implemented:

- lecturers experience source processing, blueprint generation, review, approval, and publication as one coherent workflow;
- existing backend services and frontend routes are reused rather than duplicated;
- every page recovers from backend state after refresh or restart;
- lecturer edits, revision conflicts, validation issues, and publication blockers are visible and actionable;
- publication remains explicit, immutable, version-aware, and separate from enrollment opening;
- the active published version exposes a safe, stable boundary for the next Learner Integration subsystem.
