# 38.5 — Lecturer Registration, Onboarding, and Workspace Routing

## Overview

Complete the lecturer identity and course-management subsystem by connecting registration, login, lecturer onboarding, role-aware routing, and the lecturer course dashboard into one coherent user journey.

At registration, a user chooses an initial workspace intent:

- **Student** — creates the existing learner identity and enters the learner journey.
- **Lecturer** — creates the existing learner-compatible base identity, requests the lecturer role, completes a lecturer profile, and enters the lecturer course dashboard when eligible.

After login, the server-resolved identity determines the destination:

- learner-only users go to `/dashboard`;
- active lecturer-only users go to `/lecturer/courses`;
- users with both active learner and lecturer roles go to their last authorized workspace, or a role chooser when no preference exists;
- pending, incomplete, rejected, or suspended lecturer users go to the appropriate lecturer onboarding or restricted-state page rather than receiving a generic authentication failure.

This feature must reuse Jac Scale authentication, the Feature 35 identity graph, Feature 36 authorization policy, Feature 37 course management, and Feature 38 dashboard projection. It must not introduce separate lecturer credentials, client-authoritative roles, or a second identity store.

The completed flow prepares the next subsystem—course document upload and storage—by ensuring that every upload entry point has an authenticated, active, authorized lecturer, a trusted lecturer-owned course, and a valid draft track version.

## Problem statement

The system already supports authenticated users, learner identities, lecturer profiles and roles, lecturer authorization, course creation, and a lecturer course dashboard. Those capabilities are not yet connected into a complete product flow.

Currently:

- registration creates a learner-oriented account without asking which workspace the user intends to use;
- login always redirects to the learner dashboard;
- lecturer-role request and profile completion have no dedicated UI;
- development activation requires manual private endpoint calls;
- an authenticated user can reach an `AuthGuard`-protected lecturer route without receiving a useful role-specific explanation;
- users with multiple roles have no workspace-selection model;
- the lecturer dashboard cannot reliably serve as the next step after onboarding;
- the upcoming document-upload feature has no complete UI journey proving lecturer and course ownership prerequisites.

Authentication success and lecturer authorization are different states. A valid session must not automatically grant lecturer access.

---

## Goals

Implement a complete flow that:

1. Lets a user choose Student or Lecturer during registration.
2. Keeps one shared authentication identity and credential system.
3. Creates or preserves the canonical learner-compatible identity required by existing flows.
4. Starts lecturer onboarding when Lecturer is selected.
5. Provides a dedicated lecturer profile form and status UI.
6. Supports the configured approval mode without client-side privilege escalation.
7. Resolves roles and lecturer status from the authenticated server session after login.
8. Routes users to the correct authorized workspace.
9. Supports users who hold both learner and lecturer roles.
10. Provides a persistent, non-authoritative workspace preference.
11. Protects lecturer pages with both authentication and lecturer eligibility checks.
12. Connects successful lecturer onboarding directly to `/lecturer/courses`.
13. Preserves all existing learner registration, login, assessment, roadmap, lesson, and challenge flows.
14. Produces stable error and state contracts for the frontend.
15. Verifies the complete registration-to-course-management flow.
16. Establishes the authorization and navigation prerequisites for document upload and storage.

---

## Non-goals

This feature does not implement:

- document upload, object storage, presigned URLs, or file scanning;
- document extraction or processing;
- course blueprint generation;
- course publication;
- a complete administrator dashboard;
- lecturer invitation email delivery;
- institution membership management;
- external identity providers or SSO;
- separate lecturer passwords or sessions;
- automatic production self-approval;
- destructive account or profile deletion.

The feature may add safe placeholders and route contracts for document upload, but must not accept or store document bytes.

# Design principles

## 1. Registration intent is not authorization

Choosing Lecturer at registration expresses onboarding intent. It must not itself grant an active lecturer role in production.

The server applies the configured onboarding mode:

```text
ADMIN_APPROVAL
INVITATION_ONLY
DEVELOPMENT_AUTO_ACTIVATE
```

The client must never send or persist an authoritative `is_lecturer`, `role=LECTURER`, `lecturer_id`, approval status, or capability flag.

## 2. One account may have multiple workspaces

Student and Lecturer are not mutually exclusive database account types. They are workspace choices backed by server-resolved role assignments.

An existing learner must be able to request lecturer onboarding later without creating another account.

## 3. Route from current server state

Post-authentication navigation must use a freshly loaded identity/bootstrap projection. Do not route solely from registration choice, local storage, token claims copied into application state, or a stale previous session.

## 4. Make restricted states explicit

Pending, incomplete, rejected, suspended, and archived lecturer states need distinct UI. They are not equivalent to invalid credentials or a network failure.

## 5. Resume safely

Registration, login, page refresh, browser restart, and repeated onboarding requests must resume the same lecturer profile and role assignment idempotently.

## 6. Prepare document upload at the boundary

The future upload page must be able to assume:

```text
authenticated session
active lecturer role
active lecturer profile
course view/upload permission
owned or otherwise authorized track
valid draft track version
```

The onboarding subsystem establishes the first three; course management and authorization establish the remaining prerequisites.

# User journeys

## Student registration

```text
Register
→ choose Student
→ create authentication account
→ resolve canonical learner identity and ACTIVE LEARNER role
→ establish session
→ route to /dashboard
```

The current learner onboarding behavior remains compatible.

## Lecturer registration with development auto-activation

```text
Register
→ choose Lecturer
→ create authentication account and canonical base identity
→ create/request lecturer profile and role
→ complete lecturer profile
→ server activates only when DEVELOPMENT_AUTO_ACTIVATE is configured
→ refresh identity bootstrap
→ route to /lecturer/courses
```

## Lecturer registration with administrator approval

```text
Register
→ choose Lecturer
→ create authentication account
→ complete lecturer profile
→ submit lecturer request
→ role/profile become pending
→ show /lecturer/onboarding/status
→ after server-side approval, next refresh/login resolves ACTIVE lecturer
→ route to /lecturer/courses
```

## Existing learner becomes a lecturer

```text
Login as learner
→ choose “Teach on GraphLearn”
→ open /lecturer/onboarding
→ create or resume lecturer profile
→ complete and submit
→ follow configured approval flow
```

The existing learner role and learner data remain intact.

## Login routing

```text
Login succeeds
→ load authenticated workspace bootstrap
→ evaluate server-resolved roles and lecturer state
→ route deterministically
```

Routing rules:

| Resolved state | Destination |
|---|---|
| learner active, no lecturer profile/role | `/dashboard` |
| lecturer profile incomplete/draft | `/lecturer/onboarding` |
| lecturer pending review | `/lecturer/onboarding/status` |
| lecturer active, learner not active | `/lecturer/courses` |
| learner and lecturer active, valid last workspace | last authorized workspace |
| learner and lecturer active, no preference | `/choose-workspace` |
| lecturer rejected | `/lecturer/onboarding/status` |
| lecturer suspended or archived | `/lecturer/restricted` |
| identity graph invalid | safe identity error page; no privileged workspace |

## Workspace switching

Dual-role users can switch between:

```text
Learning workspace → /dashboard
Teaching workspace → /lecturer/courses
```

The switcher stores only a preference such as `LEARNER` or `LECTURER`. The destination must be reauthorized on every use.

# Registration contract

## Client selection

Introduce a typed intent:

```jac
enum RegistrationWorkspaceIntent {
    STUDENT,
    LECTURER
}
```

The registration page must:

- present Student and Lecturer as clear, keyboard-accessible choices;
- explain that lecturer access may require profile completion and approval;
- default deliberately, not accidentally through an unchecked value;
- submit normal authentication registration first;
- start the selected application onboarding only after a valid authenticated session exists;
- prevent duplicate submission;
- recover safely if authentication succeeds but application onboarding temporarily fails.

## Failure recovery

If account creation succeeds and lecturer onboarding fails:

- do not tell the user that registration credentials failed;
- keep the authenticated session when safe;
- route to a recoverable onboarding page;
- allow idempotent retry;
- never create a second authentication account.

Recommended errors:

```text
REGISTRATION_INTENT_INVALID
IDENTITY_BOOTSTRAP_FAILED
LECTURER_ONBOARDING_START_FAILED
LECTURER_ONBOARDING_NOT_AVAILABLE
```

# Authenticated workspace bootstrap

Create one typed read contract used after registration, login, session recovery, and workspace switching.

```jac
enum WorkspaceKind {
    LEARNER,
    LECTURER
}

enum LecturerOnboardingStep {
    NOT_STARTED,
    PROFILE_REQUIRED,
    READY_TO_SUBMIT,
    PENDING_APPROVAL,
    ACTIVE,
    REJECTED,
    SUSPENDED,
    ARCHIVED
}

obj WorkspaceAvailability {
    has workspace: WorkspaceKind;
    has available: bool;
    has recommended_route: str | None;
    has reason_code: str | None;
}

obj AuthenticatedWorkspaceBootstrap {
    has user_id: str;
    has display_name: str;
    has active_roles: list[UserRoleType];
    has learner_id: str | None;
    has lecturer_id: str | None;
    has lecturer_status: LecturerProfileStatus | None;
    has lecturer_onboarding_step: LecturerOnboardingStep;
    has available_workspaces: list[WorkspaceAvailability];
    has recommended_workspace: WorkspaceKind | None;
    has recommended_route: str;
    has can_request_lecturer_role: bool;
    has generated_at: str;
}
```

Requirements:

- derive all identity and role data from the authenticated root;
- validate role/profile graph consistency;
- never accept a user ID or lecturer ID from the client;
- return safe reason codes without private graph details;
- keep the endpoint private;
- avoid returning authorization flags for unrelated resources;
- use this projection as the single post-auth routing authority.

Suggested private RPC:

```jac
def:priv load_authenticated_workspace_bootstrap() -> AuthenticatedWorkspaceBootstrap;
```

# Lecturer onboarding UI

## Routes

Add authenticated routes:

```text
/lecturer/onboarding
/lecturer/onboarding/status
/lecturer/restricted
/choose-workspace
```

Existing routes remain:

```text
/lecturer/courses
/lecturer/courses/new
/lecturer/courses/:trackId
```

## Lecturer profile form

The onboarding form consumes `LecturerProfileInput` and includes:

```text
display name                         required
professional title                  optional
institution name                    required for completion
department                          optional
areas of expertise                  at least one required
biography                           optional
staff number                        optional
```

Requirements:

- load and resume the existing profile;
- show server-provided status;
- validate required fields client-side for usability and server-side for authority;
- enforce existing length and expertise-count limits;
- normalize whitespace;
- save without creating duplicate profiles;
- distinguish Save draft from Submit when the onboarding mode supports drafts;
- disable repeated submissions while pending;
- preserve entered values after recoverable errors;
- associate errors with the relevant fields or form region;
- never expose capability fields such as `can_create_courses` for client editing.

## Status page

Display specific states:

### Pending approval

```text
Your lecturer profile is awaiting approval.
You can continue using the learner workspace while you wait.
```

### Rejected

Show a safe generic explanation and the allowed correction/resubmission action. Do not expose private administrator notes unless a later audited contract explicitly permits them.

### Suspended

Explain that teaching tools are unavailable. Do not show ordinary course-management actions.

### Active

Redirect to `/lecturer/courses` or provide a prominent “Open lecturer dashboard” action.

## Development mode

`DEVELOPMENT_AUTO_ACTIVATE` may activate a completed profile through the server onboarding service. The UI must not call an admin-only activation endpoint or simulate approval.

The existing `activate_my_pending_lecturer_for_development` endpoint must not be presented as a normal self-service action. If retained for tests or administration, its ADMIN authorization remains mandatory.

# Role-aware login and session recovery

Update login behavior:

```text
authenticate
→ persist token through the existing auth client
→ resolve current learner/application identity as required
→ load workspace bootstrap
→ navigate to bootstrap.recommended_route
```

Requirements:

- do not hard-code `/dashboard` after successful login;
- separate credential errors from post-login bootstrap errors;
- if bootstrap fails, keep a valid session and show a retry/logout recovery page;
- clear previous-account workspace state on logout;
- never inherit a previous user’s learner ID, lecturer ID, course ID, roadmap ID, or workspace choice;
- validate stored workspace preference against current server availability;
- refresh bootstrap after lecturer request, profile update, approval-state change, suspension, or role revocation;
- handle an expired token by returning to `/login` with a safe reason.

# Route protection

`AuthGuard` proves only that a session exists. Add role-aware page guards or guarded page shells.

## Lecturer guard

For ordinary lecturer management pages:

```text
session authenticated
AND active LECTURER role
AND ACTIVE lecturer profile
AND required course capability when applicable
```

Guard outcomes:

- unauthenticated → `/login`;
- draft/incomplete → `/lecturer/onboarding`;
- pending/rejected → `/lecturer/onboarding/status`;
- suspended/archived → `/lecturer/restricted`;
- active → render lecturer page.

Server endpoints must continue to enforce authorization independently. A client guard is navigation and UX, not a security boundary.

## Learner guard

Preserve existing learner behavior. A dual-role lecturer may still access learner routes when the learner role is active.

# Lecturer dashboard integration

After onboarding becomes active:

- invalidate the identity/bootstrap query;
- navigate to `/lecturer/courses`;
- load the Feature 38 authorization-scoped dashboard;
- show the lecturer display name from the server projection;
- allow course creation only when `can_create_courses` and policy checks permit it;
- expose a workspace switcher for dual-role users;
- expose profile/status access without mixing profile mutation into the dashboard projection;
- preserve dashboard search/filter/page state only within the current authenticated user session.

If a lecturer is suspended or revoked while the dashboard is open, the next endpoint request or bootstrap refresh must remove management access and route to the restricted state.

# Preparation for document upload and storage

The next subsystem will add a document action from a course dashboard/workspace. This feature must prepare the following contracts and invariants without uploading files.

## Required upload-entry projection

Course dashboard/workspace projections must identify:

```text
track_id
draft_version_id
readiness_state
actor_permissions.can_upload_document
next_action = UPLOAD_DOCUMENT when appropriate
```

## Navigation placeholder

Reserve the future route shape:

```text
/lecturer/courses/:trackId/documents
```

Until the document subsystem is implemented, the UI may show a clearly disabled or “Coming next” action. It must not accept files or claim an upload succeeded.

## Authorization handoff

The future upload service must reauthorize:

- active lecturer identity;
- `UPLOAD_COURSE_DOCUMENT` permission;
- authorized track visibility/ownership;
- draft version relationship to the track;
- course/version mutability;
- document count, size, and type policy.

No client-provided lecturer ID or owner ID may participate in this decision.

## Storage-boundary preparation

Document storage must later use opaque storage keys and safe document projections. Course dashboard and onboarding responses must never contain:

- local file paths;
- object-store credentials;
- unrestricted bucket keys;
- raw document content;
- access tokens.

# Security requirements

- Reuse the existing authentication system and JWT/session mechanism.
- Registration intent is never an authorization grant.
- Lecturer identity comes only from the authenticated server root.
- Production defaults to `ADMIN_APPROVAL` unless explicitly configured otherwise.
- `DEVELOPMENT_AUTO_ACTIVATE` must be environment-controlled and documented as non-production.
- The client cannot set role assignment status, lecturer profile status, verification status, or capabilities.
- Every lecturer course mutation reauthorizes independently.
- Route guards do not replace server authorization.
- Suspended, revoked, rejected, and archived lecturers receive no management actions.
- Workspace preferences contain no security authority.
- Logout clears account-specific client state.
- Errors do not reveal whether another user, lecturer profile, or private course exists.
- Rate-limit or otherwise bound repeated lecturer-role requests.
- Onboarding operations are idempotent.
- Audit role request, profile submission, activation, rejection, suspension, workspace selection, and denied lecturer-route access.

---

- Every form input has a visible label.

# Migration and compatibility

- Existing users remain learners and keep their current learner data.
- Existing active lecturer profiles continue to resolve as lecturers.
- Existing pending/draft lecturer profiles resume at the correct onboarding step.
- Do not duplicate lecturer profiles or role assignments.
- Do not rewrite authentication credentials.
- Treat missing workspace preference as unknown, not as learner authorization.
- Existing direct `/dashboard` and lecturer URLs remain valid when the actor is authorized.
- Existing local-storage journey data must not influence identity or workspace authorization.
- Additive DTO fields require defaults where persisted schema compatibility needs them.
