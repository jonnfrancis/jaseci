# 11-roadmap-ui.md

We're implementing the roadmap visualization UI.

This feature displays the personalized roadmap generated by `generate_roadmap`.

The roadmap UI should show the learner their course outline in a clear, structured, read-only format.

Do not generate lesson content.

Do not generate challenges.

Do not unlock lessons.

Do not update mastery.

Do not implement dashboard analytics.

---

## Goal

Create a roadmap/course outline page that displays:

* roadmap title
* roadmap description
* selected language
* estimated duration
* weeks
* lesson outlines
* milestones
* lesson generation status
* basic roadmap progress state

This UI should help the learner understand what they will study and in what order.

---

## Route / Page

Create a roadmap page.

Suggested route:

```text
/roadmap/:roadmapId
```

or use the routing convention already present in the Jac Client / React + Vite app.

The page should load a roadmap by id.

If no roadmap exists, show a helpful empty state.

---

## Data Source

Consume the roadmap data created by Feature 10.

The UI should read:

* Roadmap
* RoadmapWeek
* RoadmapLesson
* RoadmapMilestone

Do not create fake roadmap data unless used only as temporary local demo data during development.

Do not hardcode Python or Jac roadmap content in the UI.

---

## Required Roadmap Data Shape

The UI should expect roadmap data similar to:

```ts
type RoadmapView = {
  id: string;
  learner_id: string;
  language: "python" | "jac";
  title: string;
  description: string;
  estimated_weeks: number;
  status: "draft" | "active" | "completed";
  weeks: RoadmapWeekView[];
};
```

Each week should contain:

```ts
type RoadmapWeekView = {
  id: string;
  week_number: number;
  title: string;
  summary: string;
  lessons: RoadmapLessonView[];
  milestones: RoadmapMilestoneView[];
};
```

Each lesson should contain:

```ts
type RoadmapLessonView = {
  id: string;
  title: string;
  summary: string;
  target_skill_ids: string[];
  difficulty: "beginner" | "intermediate" | "advanced";
  estimated_minutes: number;
  order_index: number;
  generation_status: "pending" | "generated";
};
```

Each milestone should contain:

```ts
type RoadmapMilestoneView = {
  id: string;
  title: string;
  description: string;
  required_skill_ids: string[];
};
```

Use existing frontend DTOs if already created.

Do not create duplicate incompatible types.

---

## Page Layout

Use the existing dark LMS workspace theme.

Recommended layout:

### Header Section

Display:

* roadmap title
* roadmap description
* selected language badge
* estimated duration
* roadmap status

Example:

```text
Personalized Python Foundations Roadmap
6-week course outline generated from your assessment.
```

---

### Summary Cards

Display simple summary cards:

* total weeks
* total lessons
* total milestones
* estimated learning time

Use existing design system components.

Do not create custom card primitives.

---

### Roadmap Timeline

Display weeks in vertical order.

Each week should appear as a section/card containing:

* week number
* week title
* week summary
* lesson list
* milestone list

Weeks must be ordered by `week_number`.

Lessons must be ordered by `order_index`.

---

## Lesson Outline Display

Each lesson item should show:

* lesson title
* short summary
* difficulty badge
* estimated minutes
* generation status
* target skills

Generation status should display as:

* `pending` — lesson outline exists, full content not generated yet
* `generated` — full lesson content exists

For this feature, lesson items are read-only.

Do not link lesson cards to a workspace yet.

Do not call `generate_lesson`.

---

## Milestone Display

Each week should show milestone cards or milestone rows.

Each milestone should display:

* title
* description
* required skills

Milestones should feel like checkpoints, not graded exams.

Do not implement milestone completion logic.

---

## Progress State

Display basic roadmap progress state using existing persisted fields only.

Allowed progress display:

* roadmap status
* generated vs pending lesson count
* total weeks
* total lesson outlines

Do not infer mastery.

Do not calculate unlocked lessons.

Do not calculate learner progress from completion history yet.

Progress should be informational only.

---

## Empty States

Handle:

### No Roadmap Found

Show:

* short explanation
* CTA back to landing page or assessment start

Example:

```text
No roadmap found yet.
Complete an assessment to generate your personalized learning plan.
```

### Roadmap Not Ready

Show:

* loading skeleton
* message that roadmap generation is in progress

Do not use an indefinite spinner if the final layout is known.

---

## Loading States

Use skeleton loaders for:

* header
* summary cards
* week cards
* lesson rows

Follow the loading state guidance in `ui-context.md`.

Preserve final layout dimensions while loading.

---

## Error States

Handle:

* roadmap not found
* failed roadmap load
* malformed roadmap response
* unsupported language

Show clear retry or navigation action where appropriate.

Do not expose raw error traces to the user.

## Design Requirements

Use existing UI primitives:

* Button
* Card
* Badge if available
* ScrollArea
* Tabs only if helpful

Use Lucide icons sparingly.

Suggested icons:

* `Map`
* `Clock`
* `BookOpen`
* `Flag`
* `CheckCircle2`
* `Circle`
* `Sparkles`

Follow the existing dark-only theme.

Do not use raw Tailwind colors like `zinc-*`, `slate-*`, or hardcoded hex values.

Use tokens from `globals.css`.

* roadmap sections use semantic headings
* lesson lists are readable by screen readers
* badges are not color-only indicators
* keyboard navigation works
* focus states are visible

* week cards stack
* lesson rows remain readable
* no horizontal scrolling unless unavoidable
