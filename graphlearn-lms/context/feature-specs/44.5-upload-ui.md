# 44.5-upload-ui.md

## Overview

Implement the lecturer-facing source-material upload and processing experience that connects the complete document subsystem delivered by Features 39–44:

```text
course document upload
→ durable storage
→ security validation
→ text extraction
→ structural content chunking
→ durable processing-job state
→ ready for AI course interpretation
```

This feature turns the existing preliminary source-material form in the lecturer course workspace into a trustworthy operational interface.

The lecturer must be able to:

- select and upload one PDF or DOCX source document;
- understand the selected file before upload;
- explicitly confirm replacement of an existing source;
- observe durable processing after the upload request returns;
- understand the current stage without implementation jargon;
- recover from retryable failures;
- cancel eligible processing;
- understand when replacement—not retry—is required;
- see a concise processing timeline;
- know when the document is ready for the next subsystem;
- enter AI course interpretation only when the backend readiness guard allows it.

The core rule is:

```text
The frontend displays backend-owned document and processing projections.
It never infers readiness from filenames, elapsed time, or local state.
```

# Goals

Implement a source-material experience that:

1. Connects upload, storage, validation, extraction, chunking, and job-state contracts into one frontend workflow.
2. Keeps long-running processing independent from the upload request lifecycle.
3. Displays one backend-derived processing state and progress value.
4. Shows the current stage and completed stage history.
5. Gives the lecturer a clear next action for every terminal state.
6. Supports authorized retry from the failed stage.
7. Supports authorized cancellation where the current job allows it.
8. Makes replacement explicit and explains its consequences.
9. Prevents obsolete/replaced documents from appearing ready.
10. Resumes correctly after navigation, refresh, logout/login, or backend restart.
11. Uses safe, lecturer-readable messages rather than raw service errors.
12. Preserves the existing course-workspace information hierarchy.
13. Uses the project’s dark technical workspace design system.
14. Meets keyboard, screen-reader, reduced-motion, and responsive-layout requirements.
15. Exposes a clear readiness handoff to AI course interpretation.
16. Does not invoke byLLM or generate curriculum in this feature.

---

# Non-goals

This specification does not implement:

- document storage, scanning, parsing, extraction, or chunking logic;
- processing-job state transitions in client code;
- worker execution or scheduling in the browser;
- raw extraction-block or chunk inspection;
- AI course interpretation;
- course-blueprint generation;
- blueprint editing or approval;
- publication;
- learner-facing citations;
- multi-document course sources;
- drag-and-drop folders;
- resumable multipart uploads;
- real-time WebSockets;
- exact processing ETA;
- client-side PDF/DOCX parsing.

# Authorization

All source-workspace operations are private.

Required permissions:

```text
load status       → VIEW_COURSE_DOCUMENT
upload/replace    → UPLOAD_COURSE_DOCUMENT
retry             → PROCESS_COURSE_DOCUMENT
cancel            → PROCESS_COURSE_DOCUMENT

```

The backend must prove:

- authenticated actor;
- active lecturer role/profile;
- track ownership or authorized collaboration;
- document belongs to the requested track version;
- requested version is visible;
- mutations target an editable draft version;
- active document relationship is current.

Do not authorize from client-provided role strings or possession of `document_id`.

# Upload states

## No active document

Show a compact empty state with:

- `FileText` or `Upload` icon;
- title: `Add course source material`;
- explanation: `Upload one PDF or DOCX containing the notes GraphLearn should organize into a course.`;
- supported types;
- configured size limit;
- primary file-selection action.

Do not say the AI is reading the file before durable upload and processing actually begin.

## File selected

Show:

- filename;
- formatted size;
- detected client MIME label when present;
- remove/change selection action;
- upload action.

Client checks are preliminary usability checks only:

```text
non-empty selection
.pdf or .docx extension
declared browser MIME, when available
configured client display limit
```

The backend remains authoritative.

Do not parse the file in the browser.

## Uploading

During the HTTP upload itself:

- disable duplicate submission;
- preserve filename and size;
- show indeterminate upload progress unless real byte progress is available;
- message: `Uploading course notes…`;
- prevent navigation only if losing the current upload would actually abort it;
- do not display fake numeric upload percentage.

After a successful upload response, clear the selected browser file and load the durable processing view.

## Upload failure

Show the safe backend error and keep the file selected when retrying the same upload is safe.

Examples:

```text
The source document could not be uploaded. You can safely retry.
Only PDF and DOCX files are supported.
The selected document exceeds the configured size limit.
Another source document became active. Reload before replacing it.
```

Never show storage paths or raw exceptions.

---

# Replacement flow

Replacement is a deliberate destructive-to-current-workflow action even though historical artifacts remain retained.

Use a confirmation dialog, not an inline checkbox.

Dialog content:

```text
Replace source document?

The current processing pipeline will stop. Unapproved generated curriculum tied
to the current source may become obsolete. Historical processing records and
published curriculum are preserved.
```

The dialog must show:

- current filename;
- replacement filename;
- current processing state;
- consequences;
- `Keep current document` secondary action;
- `Replace source document` destructive/primary action.

Do not imply that published curriculum will automatically switch to the new source.

When replacement succeeds:

1. close the dialog;
2. clear previous local polling state;
3. display only the new active document;
4. begin polling the new document’s job state;
5. never merge old and new stage timelines.

# Durable processing UI

The upload request ending does not mean processing ended.

After upload, the client loads:

```text
CourseSourceWorkspaceView.processing
```

and displays its backend-owned state.

## Polling

Use bounded status polling until the backend reports a terminal or handoff-ready state.

Recommended behavior:

```text
initial interval: 2 seconds
after 30 seconds: 5 seconds
background tab: 10–15 seconds
terminal state: stop
component unmount: stop
document replacement: cancel old poll immediately
manual refresh: immediate load
```

Add small client-side jitter to polling timing if useful, but never alter backend retry scheduling.

Polling failures should not erase the last successful projection.

After a temporary read failure:

- keep the last known state;
- show `Status temporarily unavailable`;
- provide `Refresh status`;
- retry with bounded backoff.

Do not create job state in browser storage.

## Page resume

On every mount or route return:

1. load the course;
2. load the active source workspace projection;
3. render durable state;
4. resume polling only if the state is active.

Do not rely on a previous tab session to know whether processing is running.

# State-to-UI mapping

Use backend `DocumentProcessingState` as the primary mapping.

| State | Lecturer label | Message | Primary action |
|---|---|---|---|
| `UPLOADED` | Uploaded | Your file was received. | Refresh status |
| `STORED` | Stored securely | Course notes are stored and waiting to be checked. | None |
| `VALIDATION_PENDING` | Waiting to validate | Security validation is queued. | Cancel, if allowed |
| `VALIDATING` | Validating document | Checking file safety and structure. | Cancel, if allowed |
| `VALIDATION_FAILED` | Validation needs attention | This document could not pass validation. | Retry or replace, from backend policy |
| `VALIDATED` | Validation complete | The source is safe to process. | None |
| `EXTRACTION_PENDING` | Waiting to read notes | Content extraction is queued. | Cancel, if allowed |
| `EXTRACTING` | Reading course content | Extracting structured text from the notes. | Cancel, if allowed |
| `OCR_REQUIRED` | Text-based source required | This PDF does not contain enough usable text. | Replace document |
| `NO_EXTRACTABLE_TEXT` | No usable text found | Upload a text-based PDF or DOCX. | Replace document |
| `EXTRACTION_FAILED` | Reading failed | The notes could not be read. | Retry or replace |
| `EXTRACTED` | Content read | The notes are ready to be organized. | None |
| `CHUNKING_PENDING` | Waiting to organize | Course-section preparation is queued. | Cancel, if allowed |
| `CHUNKING` | Organizing course sections | Structuring the notes into reliable evidence units. | Cancel, if allowed |
| `CHUNKING_FAILED` | Organization failed | The extracted content could not be organized. | Retry |
| `CHUNKED` | Source prepared | Course sections are ready. | Continue when eligibility confirms |
| `READY_FOR_BLUEPRINT` | Ready for AI interpretation | The source is prepared for course interpretation. | Continue to course interpretation |
| `CANCELLED` | Processing cancelled | This source will not advance until processing restarts or it is replaced. | Retry/restart if allowed |
| `BLOCKED` | Action required | Processing cannot continue automatically. | Backend-projected recovery action |

Do not show enum names directly to lecturers.

# Processing progress panel

The progress panel must use:

```text
processing.summary.overall_progress_percent
processing.summary.progress_message
processing.summary.current_stage
```

Rules:

- clamp visual value to `0..100` defensively;
- never increment progress with a client timer;
- never estimate an ETA;
- display numeric percent beside the progress bar;
- keep progress monotonic for one document generation;
- only show 100% when the backend reports ready/completed;
- use cyan/in-progress treatment during processing;
- use green only for completed/ready state;
- use warning/error treatment with text for attention states.

Suggested header:

```text
Organizing course sections                         74%
[███████████████████░░░░░]
Structuring the extracted notes for reliable course interpretation.
```

Use a spinner only as a small secondary indicator for active background processing. The progress bar remains primary.

```text

```

Future AI stages must not appear as completed processing stages in this feature.

Each row includes:

- icon;
- lecturer-facing stage label;
- status text;
- attempt count only when greater than one;
- completed timestamp when useful;
- safe error indicator when failed.

Timeline states:

```text

queued
waiting
failed
cancelled
```

Do not expose `worker_id`, leases, internal attempt IDs, or raw error messages.

On narrow screens, use a vertical timeline. Do not rely on a horizontal stepper that overflows.

# Retry experience

Show retry only when:

```text
processing.summary.retry_available == true
```

The client must not maintain its own retryability table.

Retry behavior:

1. disable retry while request is in flight;
2. call the authorized retry operation;
3. replace local state with the returned projection;
4. resume polling;
5. show `Retrying from the failed step…`;
6. do not restart from upload or validation unless the backend chooses that stage.

If retry becomes unavailable between render and click, show the fresh backend state instead of treating it as a generic error.

Never offer endless retry for:

- malware or policy rejection;
- encrypted PDF;
- OCR-only/no-text documents;
- unsupported type;
- obsolete/replaced source;
- exhausted attempts.

Those states should emphasize replacement or administrator action.

---

# Cancellation experience

Show cancel only when:

```text
processing.summary.cancel_available == true
```

Use a confirmation dialog:

```text
Cancel document processing?

Completed source artifacts will remain in history, but this document will not
continue to the next processing stage.
```

Actions:

```text
Keep processing
Cancel processing
```

After request:

- display `Cancellation requested…` while the worker cooperates;
- continue polling if backend state is `CANCEL_REQUESTED`;
- stop polling at `CANCELLED`;
- do not locally force the status to cancelled.

Failure cards must answer:

1. What happened in safe terms?
2. Is the uploaded source still stored?
3. Can processing retry?
4. Must the document be replaced?
5. What should the lecturer do next?

```text
Validation is temporarily unavailable

[Retry processing] [Replace document]
```



# AI course interpretation handoff

This feature prepares—but does not execute—the next subsystem.

## Handoff definition

The AI interpretation action becomes enabled only when the backend reports:

```text
interpretation_eligible == true
AND processing.summary.state == READY_FOR_BLUEPRINT
AND active document is current
```

The frontend must not infer eligibility from:

```text
chunk_count > 0
progress == 100
processing label text
elapsed time
local upload success
```

## Ready state

Use a restrained AI-accent panel:

```text
Ready for AI course interpretation

Your source material has passed validation, been read, and organized into
traceable course sections. GraphLearn can now interpret the material into a
draft course structure.

[Continue to course interpretation →]
```

Use the AI accent only here and in future AI-owned surfaces. Earlier deterministic processing stages use brand/status colors.

## Locked state

Before readiness:

```text
Course interpretation unlocks after source processing completes.
```

Show the current blocker from the backend.

Do not render a clickable disabled-looking link. Use a true disabled button with explanatory text.

## Navigation target

Reserve a stable route:

```text
/lecturer/courses/:trackId/interpretation
```

or a curriculum-tab subroute selected by the project routing convention.

The destination may initially be a guarded placeholder owned by the next feature. Direct navigation must re-check backend eligibility and redirect safely when blocked.

## Handoff data contract

The future interpretation start request should need only trusted identifiers:

```jac
obj StartCourseInterpretationInput {
    has track_id: str;
    has track_version_id: str;
    has document_id: str;
    has request_id: str;
}
```

The backend—not the browser—must resolve:

```text
active extraction
active chunk set
chunking schema/config identity
source checksum binding
validation status
processing readiness
untrusted-content warning flags
```

Never send chunk content from the source workspace into a future AI request.

# Testing strategy

## Backend projection tests

Verify:

- authorized owner can load source workspace;
- unauthorized lecturer cannot load another lecturer’s source;
- learner cannot load lecturer processing controls;
- no-document projection returns `source_required`;
- processing view maps the active document only;
- replaced document never appears active/ready;
- retry and cancel flags come from Feature 44 projection;
- readiness uses the blueprint/interpretation guard;
- sensitive fields are absent.

## Upload UI tests

Verify:

- PDF and DOCX selection;
- unsupported extension rejected preliminarily;
- empty file rejected preliminarily;
- configured size limit displayed and checked;
- upload disables duplicate clicks;
- successful upload clears selected file;
- duplicate upload displays safe success state;
- backend rejection is displayed safely;
- refresh resumes durable processing state.

## Replacement tests

Verify:

- active document requires explicit dialog confirmation;
- cancelling dialog preserves current source;
- replacement request sets `replace_existing`;
- old poll is cancelled;
- new active document replaces old UI state;
- published provenance is not described as automatically replaced.

## Polling tests

Verify:

- active states poll;
- terminal states stop polling;
- unmount stops polling;
- route/document change stops old polling;
- temporary status error preserves last projection;
- manual refresh works;
- no duplicate polling loops are created by rerender.

## State rendering tests

Cover every `DocumentProcessingState` mapping.

Verify:

- labels are lecturer-readable;
- progress comes from backend;
- retry appears only when allowed;
- cancel appears only when allowed;
- replacement is emphasized for OCR/no-text/policy failure;
- ready state enables AI handoff;
- blocked state explains next action.

## Accessibility tests

Verify:

- file input label;
- keyboard upload/retry/cancel/replace;
- progressbar semantics;
- dialog focus behavior;
- status/alert announcements;
- no color-only states;
- disabled AI handoff explanation;
- reduced-motion behavior.

## Responsive tests

Verify desktop, tablet, and mobile layouts with:

- long filenames;
- multi-line safe errors;
- four timeline stages;
- replacement dialog;
- 200% browser zoom.

## End-to-end scenario

```text
1. Lecturer opens a draft course.
2. Source tab shows no-document empty state.
3. Lecturer selects and uploads a valid PDF.
4. Upload request completes.
5. Durable validation progress appears.
6. Status advances through extraction and chunking.
7. Refresh during extraction restores the same job state.
8. Processing reaches READY_FOR_BLUEPRINT.
9. AI interpretation handoff becomes enabled.
10. Direct navigation while not ready remains guarded.
```

Failure scenarios:

```text
scanner outage → retry available
encrypted PDF → replacement required
OCR-only PDF → replacement required
worker crash → stale recovery then progress resumes
chunking transient failure → retry from chunking
replacement during processing → old timeline stops, new document begins
```

# Acceptance criteria

## Upload

- [ ] Lecturer can select PDF or DOCX.
- [ ] Preliminary client checks improve usability without replacing backend checks.
- [ ] Upload uses the existing authorized Feature 39 boundary.
- [ ] Duplicate submission is prevented while in flight.
- [ ] Upload errors are safe and actionable.
- [ ] Browser never parses course content.

## Storage and security

- [ ] UI never exposes storage keys, paths, checksums, or credentials.
- [ ] Validation status is displayed from backend state.
- [ ] Security/policy failures show safe replacement guidance.
- [ ] Stored source remains clearly distinguished from processing success.

## Extraction and chunking

- [ ] Extraction and chunking states appear in source order.
- [ ] OCR/no-text states require replacement.
- [ ] Retryable extraction/chunking failures expose retry only when backend allows it.
- [ ] Raw extraction blocks and chunks are not exposed.
- [ ] Page provenance and chunk internals remain server-side for the next subsystem.

## Durable jobs

- [ ] Upload request may finish while processing continues.
- [ ] Refresh restores current durable job state.
- [ ] Polling stops on terminal states and unmount.
- [ ] Progress is backend-owned and bounded.
- [ ] Stage timeline uses attempt history safely.
- [ ] Retry resumes from the failed stage.
- [ ] Cancellation is cooperative and backend-confirmed.
- [ ] Replaced documents stop advancing in the UI.

## AI interpretation preparation

- [ ] AI handoff is disabled before backend eligibility.
- [ ] Ready state uses AI visual treatment and clear explanation.
- [ ] Direct navigation revalidates readiness.
- [ ] Browser does not send raw chunks to the next subsystem.
- [ ] Future interpretation can resolve the exact active chunk set server-side.
- [ ] This feature makes no byLLM call.

## UX and accessibility

- [ ] Screen answers what source is active, current progress, and next action.
- [ ] Dark tokenized UI follows `ui-context.md`.
- [ ] No hardcoded colors are added.
- [ ] Known-layout loading uses skeletons.
- [ ] Status uses text and icon, not color alone.
- [ ] Progressbar semantics are correct.
- [ ] Dialog focus and keyboard behavior are correct.
- [ ] Mobile layout has no horizontal overflow.
- [ ] Polling updates do not steal focus.

## Security

- [ ] All source operations are private.
- [ ] Backend proves lecturer and version authorization.
- [ ] Learners cannot retry or cancel course processing.
- [ ] Another lecturer cannot inspect job history.
- [ ] Fake client progress or worker identity is ignored.
- [ ] Raw exceptions and internal metadata are not rendered.

## Quality gates

- [ ] `jac check main.jac` passes.
- [ ] `jac lint main.jac` passes.
- [ ] Focused backend projection tests pass.
- [ ] Focused client component tests pass.
- [ ] Browser upload and polling QA passes.
- [ ] Accessibility checks pass.
- [ ] Existing Features 38–44 regressions pass or retain only documented unrelated blockers.
